## Identity

- **Repo:** `Lens.Core.Src`
- **Role:** Source-authoritative upstream for the LENS Workbench BMAD module.
- **Module version:** declared in [_bmad/lens-work/module.yaml](../../_bmad/lens-work/module.yaml) (`module_version`). As of this writing, `4.0.0`; re-read the manifest if you need the live value.
- **Release target:** `lens.core.release` (pulled into downstream projects as `lens.core/`).
- **Workbench state model:** Git is the only source of truth. Branches + PRs + committed artifacts = state. No database, no runtime mutable side-channel.

## The two source layers

| Layer | Path | Authored by | Role |
|-------|------|-------------|------|
| Module core (the real product) | [_bmad/lens-work/](../../_bmad/lens-work/) | Humans + agents | Skills, prompts, scripts, `lifecycle.yaml`, `module.yaml`. This is what gets promoted. |
| Adapter surface (IDE/agent entry points) | [.github/](../../.github/) plus sibling `.claude/`, `.codex/`, `.cursor/` in the release | Humans via PR | Thin wrappers exposing the module to the host environment. Some files are generated by the installer and then committed. |

**Consequence:** when a skill's behavior changes under `_bmad/lens-work/`, the matching adapter file under `.github/` usually has to change too. They ship together.

## Authority domains (inviolable)

| Domain | Path | Who may write |
|--------|------|---------------|
| Module source | `_bmad/lens-work/**` | Agents + humans in this repo, via PR |
| Adapter surface | `.github/**` | Humans only, via PR |
| Project docs | `docs/**` (this file included) | Agents + humans in this repo |
| Downstream install | `lens.core/**` (only exists in consumer workspaces) | **Nobody edits by hand — ever.** It is pulled from the release. |
| Governance repo | the separate governance repo, `main` only | **Only** the `publish-to-governance` CLI. Never direct file edits. |
| Control repo branches | `{featureId}` and `{featureId}-plan` in a control repo | Lens skills during phase work |

Cross-authority writes are hard errors. If you catch yourself about to write into `lens.core/` or a governance repo, **stop**.

## The three hard rules

1. **Do not edit `lens.core/`.** It is a read-only pulled release. Fix the source under `_bmad/lens-work/` and let promotion re-publish.
2. **Do not write directly into the governance repo.** Always call:
   ```bash
   uv run {module_path}/_bmad/lens-work/skills/bmad-lens-git-orchestration/scripts/git-orchestration-ops.py \
     publish-to-governance \
     --governance-repo {governance_repo} \
     --control-repo {control_repo} \
     --feature-id {feature_id} \
     --phase {phase}
   ```
3. **Git discipline:** `git pull` in the target repo before reading state; `git push` immediately after every commit. Full rules: [.github/instructions/lens-control-repo-git.instructions.md](../../.github/instructions/lens-control-repo-git.instructions.md).

## Control repo branch model (2-branch)

Every feature has exactly two branches in the control repo:

- **`{featureId}`** — approved base state. Updated only via PR from the plan branch.
- **`{featureId}-plan`** — planning branch. All artifact commits during active phases land here.
- Optional `{featureId}-dev-{username}` dev tracking branch.

Planning artifacts are promoted into `{featureId}` via the **plan PR** gate. Nothing else.

## State oracle

The single specification for all lifecycle behavior is [_bmad/lens-work/lifecycle.yaml](../../_bmad/lens-work/lifecycle.yaml). Phases, tracks, gates, axioms, milestones — all live there. **No phase logic is hardcoded in skills or scripts.** When you see behavior that looks like a phase rule, it came from `lifecycle.yaml`; change it there, not in code.

## Before you touch anything — preflight checklist

- [ ] `git pull` completed in this repo, no merge conflicts.
- [ ] Working directory is clean.
- [ ] You know which authority domain your edit falls under.
- [ ] You are not about to write into `lens.core/` or a governance repo.
- [ ] If editing a skill: the matching adapter in `.github/` is in scope.
- [ ] If editing `lifecycle.yaml` in a breaking way: a migration is planned (see [50-deep-dive-module.md](50-deep-dive-module.md)).
