## The 3-layer source-authoritative model

```
 ┌──────────────────────────────────────────────────────────┐
 │ Layer 3 — Adapter surfaces (.github/, .claude/,          │
 │           .codex/, .cursor/)                             │
 │   Generated by the installer, then committed.            │
 │   Thin entry points exposing the module to a host IDE.   │
 └───────────────────────────▲──────────────────────────────┘
                             │  overlay / generated from
 ┌───────────────────────────┴──────────────────────────────┐
 │ Layer 2 — Lens module source (_bmad/lens-work/)          │
 │   Skills, prompts, scripts, lifecycle.yaml, module.yaml. │
 │   Authoritative. This is what gets promoted.             │
 └───────────────────────────▲──────────────────────────────┘
                             │  promoted by CI to
 ┌───────────────────────────┴──────────────────────────────┐
 │ Layer 1 — Release distribution (lens.core.release)       │
 │   Versioned, consumer-installable bundle.                │
 │   Downstream projects pull it as lens.core/.             │
 └──────────────────────────────────────────────────────────┘
```

- Source declares `core` as a dependency and `cis`, `tea` as optional dependencies. It does **not** bundle those modules. The release bundles whatever it needs for consumers.
- Adapter files in `.github/` are authored source *products*, not scratch. They must stay aligned with the module they expose.

## Three kinds of repositories

| Repo type | Example here | Branching | Purpose |
|-----------|--------------|-----------|---------|
| Source repo | `Lens.Core.Src` (this one) | PR-driven `main` | Authors and validates the module; promotes to release. |
| Control repo | the workspace a Lens initiative runs in | 2-branch per feature (`{featureId}` + `{featureId}-plan`) | Operational workspace for active feature work. |
| Governance repo | the registry repo referenced in `_bmad/config.yaml` | `main` only, no branches | Canonical `feature.yaml`, `feature-index.yaml`, approved artifact mirror, constitutions. |

**All three are distinct.** Do not confuse the source repo (what you're editing) with a control repo (where a feature *uses* the module).

## State model: git is the only oracle

- Branch existence = initiative exists.
- PR metadata = gate decisions.
- Committed artifacts = accepted state.
- [_bmad/lens-work/lifecycle.yaml](../../_bmad/lens-work/lifecycle.yaml) is the schema for everything above.

No runtime mutable state. No git-ignored cache that tools rely on for correctness. If a piece of state isn't committed, it doesn't exist.

## The 4-level constitution hierarchy

Governance rules are composed from four levels, **additive** (lower levels add, they can never remove):

```
org  →  domain  →  service  →  repo
```

- Each level is a YAML file in the governance repo under `constitutions/`.
- The [bmad-lens-constitution](../../_bmad/lens-work/skills/bmad-lens-constitution/SKILL.md) skill resolves and merges the stack at skill-invocation time.
- Language-specific constitutions apply only when the target repo's declared language matches.

## Promotion flow (source → release)

1. PR merges to `main` in `Lens.Core.Src`.
2. `.github/workflows/promote-to-release.yml` triggers on paths under `_bmad/lens-work/**`, `.github/agents/**`, etc.
3. The workflow:
   - Reads `module_version` from [_bmad/lens-work/module.yaml](../../_bmad/lens-work/module.yaml).
   - Validates the source (registry check, preflight).
   - Runs a clean BMAD install.
   - Overlays `_bmad/lens-work/` and executes the installer (`_module-installer/installer.js`).
   - Overlays `.github/` and validates alignment.
   - Commits to `lens.core.release` on the appropriate branch (alpha / beta / stable per the release branch model).
4. Downstream consumers pull the release and get `lens.core/` populated.

Local pre-promotion checks live in [40-task-recipes.md](40-task-recipes.md#trigger-or-dry-run-a-release).

## Why this shape

- **Separation of authoring and packaging.** Authors never touch the distributed form; they touch primitives. CI owns the bundling.
- **Git-derived state** keeps the system debuggable: every agent action leaves a branch, a commit, or a PR.
- **Additive constitutions** let organizations layer guardrails without forking skills.
- **Two-layer source (module + adapter)** keeps the adapter surface small and swappable per host IDE, while the module stays host-agnostic.
